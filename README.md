# Requirement Analysis Skill

> WorkBuddy 技能 — 从模糊想法中挖掘真需求，支持批量需求识别与逐个深挖

## 简介

这是一个 [WorkBuddy](https://www.codebuddy.cn/docs/workbuddy/Overview) 技能（Skill），帮助用户通过结构化提问从口语化描述、文档或大段文本中挖掘真正的需求。

**核心理念：** 需求的本质 = 目标（To-Be）与现状（As-Is）之间的差距。用户提出的往往是缩小差距的"某个方案"，而这个技能的作用是帮用户回到差距本身，找到"真正要解决的问题"。

### 与普通 AI 对话的区别

| 维度 | 普通 AI 对话 | 本 Skill |
|------|-------------|---------|
| 需求识别 | 用户说什么就做什么 | 区分"需要"与"方案"，挖掘真需求 |
| 信息收集 | 一次性问一堆问题 | 每轮只问一个，逐维度递进 |
| 多需求处理 | 逐条单独处理 | 先批量识别列出清单，再逐个深挖 |
| 输出格式 | 自由格式 | 结构化用户故事 + As-Is/To-Be 对比表 |
| 汇总能力 | 无 | 多需求优先级、依赖关系、冲突提示 |

## 安装

### 方式一：手动安装

1. 下载本仓库的 `requirement-analysis/` 目录
2. 将整个目录复制到 WorkBuddy 的技能目录：
   ```
   # 用户级（所有项目可用）
   ~/.workbuddy/skills/requirement-analysis/
   
   # 项目级（仅当前项目可用）
   .workbuddy/skills/requirement-analysis/
   ```
3. 重启 WorkBuddy，技能将自动生效

### 方式二：通过打包文件安装

1. 下载 `requirement-analysis.zip`
2. 解压到上述技能目录
3. 重启 WorkBuddy

### 验证安装

在 WorkBuddy 对话中输入：
> 帮我分析需求

如果技能被正确加载，WorkBuddy 会进入需求分析模式。

## 使用方法

### 场景一：单需求分析

直接在对话中描述你的想法或需求：

```
用户：我想做一个每日打卡 App
AI：我理解你的意思是想做一个每日打卡的应用，对吗？
    那我们先从最基础的问题开始——在这个需求中，你扮演的是什么角色？
...
```

AI 会依次通过四个维度追问：
1. **角色** — 你是谁？
2. **场景** — 什么时候、在哪里？
3. **动作** — 你要做什么？
4. **期望** — 为什么要做？做到什么程度算成功？

追问完成后，AI 会输出一份结构化需求文档。

### 场景二：批量需求分析

上传文档或粘贴大段文本，AI 会先识别所有需求：

```
用户：[上传一份产品需求文档 / 粘贴一大段需求描述]

AI：📋 识别到以下 5 个需求：
    1. 【每日打卡】— 用户可以记录每日完成情况
    2. 【数据统计】— 查看打卡历史和连续天数
    3. 【好友PK】— 与好友比拼打卡连续天数
    4. 【提醒推送】— 定时提醒用户打卡
    5. 【自定义目标】— 用户可以创建自己的打卡目标
    
    请确认：
    - 哪些需求需要深入分析？（回复序号，如"1,3,5"或"全部"）
    - 是否有遗漏或需要补充的需求？

用户：全部

AI：好的，现在开始逐个分析。
    正在分析第 1 个需求（共 5 个）：【每日打卡】
    在这个需求中，你扮演的是什么角色？
...
```

所有需求分析完成后，AI 会输出一份汇总表，包含优先级、依赖关系和冲突提示。

## 工作流程

```
Phase 0: 批量需求识别（仅批量模式）
  ├─ 读取文档/文本
  ├─ 筛选并列出需求清单
  └─ 用户确认分析范围
      │
      ▼
Phase 1-4: 逐个深挖（每个需求独立执行）
  ├─ Phase 1: 接收需求，复述确认
  ├─ Phase 2: 四维度追问（角色→场景→动作→期望）
  ├─ Phase 3: As-Is/To-Be 差距分析 + 方案vs需要剥离
  └─ Phase 4: 输出结构化需求文档
      │
      ▼
Phase 5: 多需求汇总（仅批量模式）
  ├─ 需求清单表（标题/真需求/优先级/状态）
  ├─ 依赖关系分析
  └─ 冲突提示
      │
      ▼
Phase 6: 收尾确认
```

### 四维度追问详解

| 维度 | 核心问题 | 目标 |
|------|---------|------|
| 角色 | 你是谁？ | 明确需求提出者的身份和上下文 |
| 场景 | 什么时候、在哪里？ | 明确触发情境和当前处理方式 |
| 动作 | 你要做什么？ | 明确期望行为和与现状的差距 |
| 期望 | 为什么要做？做到什么程度？ | 挖掘真正动机，区分方案与需要 |

### 追问规则

- **每轮只问一个问题**，等用户回答后再问下一个
- **按维度顺序推进**：角色 → 场景 → 动作 → 期望，不跳跃
- **自动跳过已覆盖的问题**：如果用户在前面的回答中已经提到了后续维度的信息，不重复追问
- **期望维度的"关键问题"必问**：你说的做法是唯一方式吗？有没有替代方案？

## 输出格式

### 单需求文档

```markdown
# Requirement: [一句话标题]

## 1. Overview
[2-3 句话：谁，在什么情况下，需要什么，为了什么目的]

## 2. User Story
> As a **[角色]**,
> when **[场景]**,
> I want to **[动作]**,
> so that **[期望/目的]**.

## 3. As-Is / To-Be Comparison
| 维度 | 现状 (As-Is) | 目标 (To-Be) |
|------|-------------|-------------|
| ...  | ...         | ...         |

## 4. Acceptance Criteria
- [ ] [可衡量的标准]

## 5. Constraints & Boundaries
- [约束条件]

## 6. Alternative Approaches
- 原始方案：[用户最初提的方案]
- 替代思路：[更优方案]
```

### 多需求汇总表

```
📊 需求分析总结（共 N 个需求）

| # | 需求标题 | 真需求 | 优先级 | 状态 |
|---|---------|--------|--------|------|
| 1 | ...     | ...    | 高     | 已完成 |
| 2 | ...     | ...    | 中     | 已完成 |

依赖关系：
- 需求 2 依赖于需求 1 的完成

冲突提示：
- 需求 A 和需求 B 在资源分配上存在冲突
```
